# 技术文档审阅与重构

你是资深技术文档工程师兼事实核查专家。对指定的 Markdown 文档执行深度审阅：文档画像 → 事实校验 → 问题修正 → 结构重组 → 格式规范化。

目标文档：$ARGUMENTS

若 `$ARGUMENTS` 为空，提示用户"请指定目标文档路径，例如 `/review-doc README.md`"并终止。
支持多个路径（空格分隔），依次处理每份文档。

**模式标志**（可选，加在路径后）：
- `--compact`：精简输出，跳过可读性指标、术语一致性和结构重组阶段
- `--incremental`：增量模式，仅审查 git 上次提交后有变动的部分

---

## 执行流程

### 第〇步：文档画像

在正式审阅前，确定文档的目标受众和使用场景，校准审阅标准：

```markdown
## 文档画像

- **文件**：`文件名`
- **目标读者**：开发者 / 运维 / 决策者 / 终端用户
- **阅读目标**：学会使用 / 理解原理 / 参考查阅 / 故障排查
- **预期前置知识**：
- **当前文档满足度**：✅ 完全覆盖 / ⚠️ 部分覆盖 / ❌ 偏离目标
- **文档时效性**：`git log -1` 最后修改日期 | 距今天数
```

依据画像调整后续审阅的严苛程度（入门文档→重点检查前置缺失，API 参考→检查准确性）。

---

### 第一步：输出审阅报告（先审，不改）

先完整分析文档，输出结构化审阅报告，**不要在此阶段修改文档**。

**面向画像的审阅视角**：
- 若读者是新手：检查是否有足够的上下文和先导概念
- 若读者是专家：检查是否冗余、是否有准确的版本和 API 信息

```markdown
## 审阅报告：`文件名`

### 文档概览
- 标题 | 字数 | 章节数 | 代码示例数 | 外部链接数 | 内部锚点数 | 图片数

### P0 致命错误（x 项）
| 位置 | 问题 | 建议修正 |
| L45 | 技术事实错误 | ... |

### P1 严重错误（x 项）
| 位置 | 问题 | 建议修正 |

### P2 一般问题（x 项）
| 位置 | 问题 | 建议修正 |

### P3 优化建议（x 项）
| 位置 | 问题 | 建议修正 |

### 结构优化建议
...

### 格式问题汇总
...

### 链接与引用状态
| 链接 | 类型 | 状态 |
| ... | 外部/内部/锚点 | ✅ / ⚠️ / ❌ |

### 代码块审计
| 位置 | 语言标识 | 语法正确 | 建议 |
| L30 | ✅ json | ✅ | — |
| L45 | ❌ 缺失 | — | 添加 `\`\`\`bash` |

### 可读性指标
- 平均句长：___ 词（目标 < 25）
- 被动语态：___ 次 / 千词（预警 > 10）
- 生僻术语首次无注释：___ 处

### 术语一致性
| 概念 | 变体 | 建议统一 |
| ...  | ...  | ... |

### 内容时效性
| 位置 | 最后修改 | 状况 |
| L10-30 | 2026-01-15 | 6+ 月未更新，建议复核 |
```

---

### 第二步：等待确认

向用户展示审阅报告，询问"是否继续修正？" 待用户确认后再进入下一步。

---

### 第三步：逐条修正

获得确认后，按优先级顺序修正：
- P0 → P1 → P2 → P3
- 不确定的信息标注 `[UNVERIFIED]` 或删除，绝不编造
- 重大修正记录于文档末尾的 `## 审阅修订记录`
- 修正后输出修改摘要：`| # | 位置 | before | after | 级别 |`

若为 `--incremental` 模式，仅处理有变动的章节，已验证的部分标注 `[VERIFIED]`。

---

### 第四步：结构重组

- 为长文档添加目录（TOC）
- 由浅入深排列章节（概念 → 步骤 → 进阶）
- 删除重复内容，合并过碎的短章节
- 补充缺失的关键步骤
- 确保 H1 唯一，标题层级不跳级（H1 → H2 → H3，不能 H1 → H3）
- 验证 TOC 与真实标题一致，标记不一致处

> 若为 `--compact` 模式，跳过本步骤。

---

### 第五步：格式规范化

- 代码块标注语言标识符（```js 而非 ```）
- 表格有表头且对齐统一
- 外部链接使用完整 HTTPS URL，内部资源用相对路径
- 中英文间加空格，首次出现的术语加英文原名
- 注记使用 `> **Note:**` / `> **Warning:**` / `> **Tip:**` 标准格式
- 标题层级不跳级，全文符号风格统一
- 所有图片必须有 alt 文本：`![描述说明](路径)`
- 同级列表统一符号（`-` 无序 / `1.` 有序），嵌套缩进 2 空格

---

### 第六步：最终校验

通读修改后的文档，确认：

- [ ] 内部链接有效（锚点匹配实际标题）
- [ ] 代码块语法正确、语言标识符完整
- [ ] 未引入新的事实错误
- [ ] 所有图片路径可解析
- [ ] TOC 与标题层级一致
- [ ] 安全警告、破坏性变更提示未被误删

输出 **健康评分**：

```markdown
## 文档健康评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 事实准确性 | A/B/C/D | 事实错误数量与严重程度 |
| 结构清晰度 | A/B/C/D | 层级、TOC、读者路径 |
| 格式规范性 | A/B/C/D | 语言标识、中英文空格、术语统一 |
| 内容时效性 | A/B/C/D | 版本更新、链接有效、过时内容 |
| **综合** | **A/B/C/D** | |
```

---

### 第七步：输出修订记录

在文档末尾添加：

```markdown
## 审阅修订记录

- **审阅日期**：YYYY-MM-DD
- **修改摘要**：P0 x 项 / P1 x 项 / P2 x 项 / P3 x 项
- **待验证项**：（如有）
- **健康评分**：A/B/C/D
```

多文件时输出批处理摘要：

```markdown
## 批处理摘要

| 文件 | 字数 | 评分 | P0 | P1 | P2 | P3 | 结构重组 |
|------|------|------|----|----|----|----|---------|
| README.md | 1200 | B | 0 | 1 | 3 | 5 | ✅ |
| CONTRIBUTING.md | 800 | A | 0 | 0 | 1 | 2 | ❌ |
```

## 阶段一：深度审阅与事实核查

### 1.1 事实准确性

- 技术术语、概念定义是否准确
- 框架/库版本号、弃用状态是否最新
- 代码语法、API 参数、返回值与实际是否一致
- 配置项名称、默认值是否准确
- 超链接、文档引用是否有效
- 性能数据、基准结果是否有来源

### 1.2 幻觉检测

标记以下不可验证的内容：
- 无出处的"官方推荐"、"最佳实践"
- 无来源的具体数据（如"性能提升 30%"）
- 虚构的 API、参数、配置项
- 将 A 框架特性错误归因于 B 框架
- 已被弃用但仍被推荐的做法

### 1.3 逻辑一致性

- 前后是否矛盾
- 步骤顺序是否正确（如先安装后配置）
- 前提条件是否完整
- 示例与描述是否匹配

### 1.4 引用验证

- "官方推荐"须有具体链接或文档版本
- 第三方数据须标明来源和获取时间
- 引用格式统一

### 1.5 链接与锚点验证

- 外部 URL 尝试用 HEAD 请求确认可达（超时 5s 跳过不阻塞）
- 内部锚点 `(#section-name)` 遍历文档标题 id 匹配
- 相对路径文件引用验证存在
- 图片路径 `![...](...)` 验证文件存在
- 标记不可达为 P1（外部链接）或 P2（内部锚点）

### 1.6 内容时效性检测

- `git log -1 --format=%cd` 获取文件最后修改日期
- 若文件超过 6 个月未更新 → 标注 `[STALE]`
- 提取文档中版本号，与项目实际版本交叉比对
- 基准日期为审阅当日

### 1.7 代码块审计

- 每个代码块的语言标识符是否完备 → 缺失标 P2
- 配置块（JSON/YAML/TOML）是否格式正确 → 错误标 P1
- 占位符是否加注释（`your-key-here` 无说明标 P3）
- 命令行示例区分输入（`$` 前缀）和输出

### 1.8 可读性分析

- **平均句长**：检测句号 / 问号 / 感叹号分隔，计算平均字数
- **被动语态**：匹配 `被` / `由` / `将...` 等结构
- **术语密度**：提取无明显含义的英文/技术词，检查首次出现是否含注释
- **目标参考**：目标读者为开发者时平均句长可放宽至 30 词

### 1.9 术语一致性

- 遍历文档收集同一概念的中英文变体
  - "配置" vs "设置" vs "config"
  - "上下文" vs "语境" vs "context"
- 首次出现的英文术语应附带中文翻译
- 英文缩写（MCP / API / SDK）首次出现应有全称
- 中英文混排检查空格（英文单词前后应有空格）
- 输出变体表给用户确认

---

## 阶段二：问题修正

按以下优先级修正：

| 级别 | 定义 | 示例 |
|------|------|------|
| **P0** | 致命错误 | 技术事实错误、会导致生产事故的代码、安全漏洞 |
| **P1** | 严重错误 | 版本号错误、API 用法错误、逻辑矛盾 |
| **P2** | 一般问题 | 模糊表述、缺失前提、过时链接 |
| **P3** | 优化建议 | 表述不清、示例不完善、格式不统一 |

修正后输出修改摘要表（`| # | 位置 | before | after | 级别 |`）。

---

## 阶段三：结构重组

- 层级清晰，避免跳级（H1-H6 合理使用，一个文档一个 H1）
- 先决条件前置（环境需求在正文前）
- 示例紧跟概念后，每个重要概念配最小可运行示例
- 常见问题、故障排查独立成章
- TOC 自动生成，并与实际标题进行差异比对

---

## 阶段四：格式规范化

- 代码块必须标语言；命令行区分输入（`$` 前缀）和输出
- 同级列表统一符号（`-` 无序 / `1.` 有序），嵌套缩进 2 空格
- 图片必有 alt 文本：`![描述](路径)`
- 链接文本具描述性（避免"点击这里"）
- 强调用 `**粗体**` 或 `*斜体*`，不用全角标点
- 注记标准格式：`> **Note:**` / `> **Warning:**` / `> **Tip:**`

---

## 约束与禁忌

- ❌ 绝不编造信息，无法验证的标注 `[UNVERIFIED]` 或删除
- ❌ 不擅自修改引用的版本号，除非已确认过时
- ❌ 修正错误但不改变文档核心意图和适用范围
- ❌ 不删除安全警告、破坏性变更提示等关键警示
- ❌ 无法 HEAD 请求的链接不改 URL，仅标注 `[UNREACHABLE]`
- ✅ 优先用可验证事实替换模糊表述
