# /kb/status — 生成状态报告

## 前置检测

读取 `.kb/kb.json`：
- 如果文件不存在，提示"请先运行 /kb/init 初始化知识库"
- 如果文件存在但格式损坏，提示错误并建议重新初始化

---

## Step 1：选择报告类型

使用 `question` 工具：

```
问题：「要生成什么类型的报告？」
选项：
  [1] 📊 全面报告 — 完整的知识库健康检查（推荐定期使用）
  [2] 📈 进度摘要 — 仅进度和阻塞项
  [3] 🔗 概念图谱 — 概念间的依赖关系图
  [4] ⚠️ 薄弱环节 — 诊断框架薄弱点
  [5] ❌ 返回
```

---

## Step 2：收集数据

### 2.1 读取项目元数据

读取 `.kb/kb.json`：
- name, type, phase
- created, version

### 2.2 统计文档

使用 `glob` 工具统计 `.kb/docs/` 各子目录的文件数：

```
docs/decisions/  → .kb/docs/decisions/*.md
docs/notes/      → .kb/docs/notes/*.md
docs/progress/   → .kb/docs/progress/*.md
docs/specifications/ → .kb/docs/specifications/*.md
```

### 2.3 读取索引

读取三个索引文件：
- `.kb/index/concepts.json` — 概念统计
- `.kb/index/questions.json` — 问题统计
- `.kb/index/timeline.json` — 时间线分析

### 2.4 读取上下文

读取 `.kb/context/state.md` 和 `.kb/context/project.md` 获取最新状态和项目信息。

### 类型 3：🔗 概念图谱

```
🔗 概念依赖图谱
═══════════════════════════════════════════════

【基础层 — 核心概念】

  误差信号 🟢
    定义：系统输出与期望输出之间的偏差
    操作化：|actual - expected|
    上游：无
    下游：Lyapunov函数, 控制器设计

  状态空间 🟡
    定义：系统所有可能状态的集合
    操作化：向量维度 + 采样精度
    上游：无
    下游：PID控制器, 轨迹规划

【中间层 — 理论工具】

  Lyapunov函数 🟡
    定义：正定标量函数，导数负定则系统稳定
    操作化：需定义语义距离度量
    上游：误差信号
    下游：元认知约束层

  PID控制器 🟢
    定义：比例-积分-微分控制
    操作化：K_p, K_i, K_d 参数
    上游：状态空间, 误差信号
    下游：元认知约束层

【应用层 — 最终目标】

  元认知约束层 🔴
    定义：监控并约束Agent行为发散
    操作化：待完成
    上游：Lyapunov函数, PID控制器, 信息论约束
    下游：无

═══════════════════════════════════════════════

悬空概念（需尽快操作化）：
  ⚠️ 语义空间距离度量
     → 阻塞：Lyapunov函数操作化
  ⚠️ 信息论约束
     → 阻塞：元认知约束层设计
```

### 类型 4：⚠️ 薄弱环节

```
⚠️ 框架薄弱环节诊断
═══════════════════════════════════════════════

【最薄弱环节】

  🔴 语义空间距离度量
     位置：基础层
     问题：无法在语义空间中定义"距离"
     影响：Lyapunov函数无法操作化，整个稳定性证明受阻
     修复建议：
       1. 调研语义空间中距离度量的已有研究
       2. 考虑用余弦相似度或欧氏距离的变体作为代理
       3. 明确距离度量的失效条件

【次要薄弱环节】

  🟡 PID控制器离散动作空间适配
     位置：中间层
     问题：经典PID面向连续系统，Agent动作离散
     影响：控制器设计需要近似
     修复建议：
       1. 调研离散PID或Bang-Bang控制
       2. 评估近似引入的误差是否可接受

【架构风险】

  ⚠️ 元认知约束层独立性假设
     问题：假设元认知层可独立于执行层
     风险：如果耦合过紧，独立性不成立
     建议：明确耦合边界，定义接口规范

═══════════════════════════════════════════════
```

## Step 4：更新状态摘要

执行报告生成后，自动更新 `.kb/context/state.md`：

```markdown
# 当前状态摘要

最后更新：{ISO 8601 日期时间}

## 进度摘要
（根据报告内容自动生成，限制 200 字）

## 活跃概念
（从 concepts.json 中提取 status != "archived" 的概念）

## 待解决的高优先级问题
（从 questions.json 中提取 priority == "高" 且 status == "open" 的问题）
```

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| kb.json 不存在 | 提示"请先运行 /kb/init 初始化知识库" |
| index 文件部分缺失 | 仅处理存在的文件，缺失部分标注"数据不可用" |
| concepts.json 损坏 | 提示"概念索引损坏"，跳过概念统计部分 |
| state.md 更新失败 | 提示警告但不中断，报告仍可正常生成 |
| docs 目录为空 | 提示"知识库为空，建议使用 /kb/update 添加内容" |

---

## 与其他命令的数据链路

```
依赖：
  /kb/init — 提供 kb.json（项目信息）
  /kb/update — 写入 docs/               ← 文档统计
               index/concepts.json        ← 概念统计
               index/questions.json       ← 问题统计
               index/timeline.json        ← 趋势分析
               context/state.md          ← 状态读取
               context/project.md         ← 项目读取

产出：
  /kb/status → context/state.md 更新    ← /kb/init (恢复) 加载
               状态报告（对话内展示）
```
