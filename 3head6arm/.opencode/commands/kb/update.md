# /kb/update — 添加 / 更新知识

## 前置检测

读取 `.kb/kb.json`：
- 如果文件不存在，提示"请先运行 /kb/init 初始化知识库"
- 如果文件存在但格式损坏，提示错误并建议重新初始化

---

## Step 1：选择更新类型

使用 `question` 工具：

```
问题：「这次要记录什么类型的内容？」
选项：
  [1] 📝 学习笔记 — 记录新学到的概念、理论或技术
  [2] 🔧 决策记录 — 记录一个设计/架构决策及其理由
  [3] 📊 进度更新 — 记录当前阶段的进展和状态
  [4] 📐 规格文档 — 记录系统/接口的正式规格
  [5] ✏️ 修改已有 — 补充或修正已有的知识条目
  [6] ❌ 取消
```

---

## Step 2：根据类型引导内容

### 类型 1：📝 学习笔记

**A. 主题**

```
问题：「这个笔记的主题是什么？（简短标题）」
示例：「Lyapunov 稳定性理论」「REST API 认证机制」
```

**B. 领域（可选）**

```
问题：「它属于哪个领域/分类？用于索引，可跳过。」
示例：「控制理论」「前端架构」「机器学习」
```

**C. 核心要点**

```
问题：「用几句话总结你学到了什么。」
要求：不要照搬资料，要用自己的语言表述
```

**D. 与项目关联**

```
问题：「这个知识点如何服务于当前项目？」
示例：「为'语义稳定性'提供数学形式化工具」「可能适用于用户认证模块」
```

**E. 学以致用引导**

```
问题：「是否需要学以致用引导？」
选项：
  [1] 是，帮我做操作化检验 — 深入分析概念的可测量性
  [2] 否，仅记录
```

---

### 类型 2：🔧 决策记录

**A. 决策主题**

```
问题：「这次记录什么决策？」
示例：「选用 Lyapunov 直接法判断稳定性」「采用 JWT 而非 Session」
```

**B. 背景**

```
问题：「决策的背景是什么？要解决什么问题？」
```

**C. 考虑过的选项**

```
问题：「考虑过哪些方案？简要说明每个方案的核心权衡。」
提示：至少列出 2 个选项，如果只有一个方案则不适合用决策记录
```

**D. 最终选择**

```
问题：「最终选择了哪个方案？选择它的核心理由是什么？」
```

**E. 后果**

```
问题：「这个决策会带来什么后果？预计会产生什么影响？」
提示：包括正面后果和需要关注的潜在风险
```

---

### 类型 3：📊 进度更新

**A. 更新周期**

```
问题：「这次更新覆盖什么时间段？」
示例：「2025-06-01 至 2025-06-15」「本周」「 Sprint 3」
```

**B. 本期成果**

```
问题：「本期完成了什么？」
要求：列出具体成果，最好可量化
```

**C. 阻塞项**

```
问题：「有什么阻塞了进展？」
选项：
  [1] 无阻塞
  [2] 有阻塞 — 说明原因
```

**D. 下步计划**

```
问题：「下一步计划做什么？」
```

**E. 需要的帮助**

```
问题：「需要什么帮助或资源？可跳过。」
```

---

### 类型 4：📐 规格文档

**A. 规格名称**

```
问题：「这个规格文档的名称是什么？」
示例：「误差信号接口规范」「前端组件编写标准」
```

**B. 规格类型**

```
问题：「这是什么类型的规格？」
选项：
  [1] 接口规格 — API、模块间的输入输出约定
  [2] 数据规格 — 数据结构、格式、编码约定
  [3] 流程规格 — 业务流程、操作步骤约定
  [4] 质量规格 — 性能、安全、可访问性等要求
```

**C. 适用范围**

```
问题：「这个规格适用于什么范围？」
示例：「所有误差信号计算函数」「所有对外 API」
```

**D. 详细描述**

```
问题：「详细描述这个规格的内容。」
提示：接口规格应包含方法签名、数据格式、错误处理等
```

**E. 示例**

```
问题：「有示例吗？可选，但强烈建议提供。」
```

---

### 类型 5：✏️ 修改已有

**A. 选择要修改的文档**

使用 `question` 工具展示 timeline：

```
问题：「选择要修改的文档」
说明：从最近的时间线展示文档列表，用户选择
```

读取 `.kb/index/timeline.json`，展示最近 10 条（排除已归档），格式：

```
[1] YYYY-MM-DD 标题（类型标签）
[2] YYYY-MM-DD 标题（类型标签）
...
[0] 返回上一步
```

**B. 修改类型**

```
问题：「要做什么修改？」
选项：
  [1] 补充内容 — 在现有基础上添加新信息
  [2] 修正错误 — 修正已有内容中的错误
  [3] 标注状态 — 更新操作化状态、优先级等元数据
```

**C. 补充内容 / 修正内容**

```
问题：「补充内容是什么？」或「正确的表述是什么？」
```

## Step 4：执行写入

### 4.1 确定文件名

```
格式：YYYY-MM-DD-<slug>.md
slug 生成规则：
  - 英文：转小写，替换空格为连字符，去除非字母数字字符
  - 中文：直接使用，保留中文字符
  - 冲突检测：如果文件已存在，追加 -2, -3 ...
```

### 4.2 写入文档

根据类型使用对应模板生成 Markdown，写入 `.kb/docs/<type>/`。

**学习笔记模板：**

```markdown
---
date: "{ISO 8601}"
type: note
title: "{title}"
domain: "{domain}"
tags: [{tags}]
---

# {title}

## 核心要点
{user_content}

## 与项目的关联
{project_relevance}

## 操作化状态
{operationalization_status}

## 关联概念
{related_concepts}

## 待解决问题
{open_questions}

## 相关文档
{related_docs}
```

### 4.3 更新索引

**concepts.json 更新逻辑：**

如果文档涉及新概念，从文档中提取概念名，检查 `concepts.json` 中是否已存在：
- 不存在：追加新概念节点
- 已存在：更新概念信息

概念节点结构：
```json
{
  "name": "概念名称",
  "status": "🟢 已操作化 | 🟡 待操作化 | 🔴 悬空",
  "operationalization": "如何被测量/计算",
  "source_doc": "docs/notes/YYYY-MM-DD-slug.md",
  "related": ["其他概念名"],
  "created": "{ISO 8601}",
  "updated": "{ISO 8601}"
}
```

**questions.json 更新逻辑：**

如果文档标记了待解决问题，追加到 `questions.json`：
```json
{
  "id": "q-{auto-increment}",
  "description": "问题描述",
  "source_doc": "docs/notes/YYYY-MM-DD-slug.md",
  "priority": "高 | 中 | 低",
  "blocked_by": [],
  "created": "{ISO 8601}",
  "status": "open"
}
```

**timeline.json 更新逻辑：**

追加新条目：
```json
{
  "date": "{ISO 8601}",
  "type": "{note|decision|progress|spec|update}",
  "description": "{title}",
  "path": "docs/[type]/YYYY-MM-DD-slug.md",
  "archived": false
}
```

## Step 5：学以致用引导（如果用户选择）

当用户选择「是，帮我做操作化检验」时，对概念执行操作化检验：

### 操作化三问

**问题一：如何被计算？**

```
在你的系统场景中，这个概念的精确计算方式是什么？
- 输入变量是什么？从哪里获取？
- 计算公式或算法是什么？
- 输出是什么量纲/范围？
```

**问题二：如果无法直接计算**

```
最优的代理指标（proxy metric）是什么？
- 与原概念的相关性有多强？
- 代理指标的已知偏差在哪里？
- 什么情况下代理指标会失效？
```

**问题三：测量方案的失效条件**

```
这个测量方案在什么情况下会给出错误信号？
- 误报（false positive）：实际没问题但报警
- 漏报（false negative）：实际有问题但未报警
- 对你的控制系统设计意味着什么风险？
```

**工程可行性评级：**

```
评级：
🟢 直接可工程化 — 有明确计算方法
🟡 需要近似 — 代理指标可用，但有已知误差
🔴 当前不可操作化 — 需要进一步理论工作才能落地

如果评级为🔴，说明需要解决什么理论问题才能让它变成🟡或🟢。
```

## Step 6：输出确认

```
✅ 已保存。
📄 1 个文档 | 🧠 {N} 个新概念 | ⚠️ {M} 个待解决问题

文档路径：docs/[type]/YYYY-MM-DD-slug.md
```

如果创建了新概念或待解决问题，显示简要列表。

---

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| kb.json 不存在 | 提示"请先运行 /kb/init 初始化知识库" |
| 用户取消 | 输出"已取消"，不产生任何文件 |
| 目录创建失败 | 提示权限错误 |
| 文件写入失败 | 提示错误并说明原因 |
| 索引 JSON 损坏 | 尝试读取备份（如有），否则提示重建索引 |

---

## 与其他命令的数据链路

```
依赖：
  /kb/init — 提供 kb.json（项目信息）
  /kb/init — 提供 index/concepts.json（已有概念列表，用于关联）
  /kb/query — 读取概念列表供用户选择关联

产出：
  /kb/update → docs/[type]/*.md    ← /query 全文搜索 / /archive 归档
               index/concepts.json  ← /query /status 读取
               index/questions.json ← /query /status 读取
               index/timeline.json   ← /query /status 读取
```
